Lesson 10 — State Machine

An explicit state machine makes impossible states unreachable. From success you can't go back to loading without going through reset to idle first. The transition table in machine.js is the contract: if an event isn't listed for the current state, next() warns and returns the current state unchanged — the UI simply cannot get into an undefined configuration.

Open the browser console and try clicking "Fetch (succeeds)" while already in success state — you'll see the invalid-transition warning, and nothing will break.

Note: the initial fetch is deterministic (you pick "succeeds" or "fails"), but Retry randomizes the outcome — sometimes you'll land back in error. That's intentional: real-world retries don't come with a guarantee, and the state machine has to handle either result.

What's next — Lesson 11 (Robustness)

Lesson 11 (Robustness) used an implicit state machine: the <user-card> component tracked idle → loading → success | error via data-state attributes and method calls, with no single source-of-truth for which transitions were legal.

This lesson is the formal version of that same idea — the transition table in machine.js makes every valid move explicit and every invalid move auditable. Same concept, made testable: you can import next() and assert on transitions without touching the DOM at all.

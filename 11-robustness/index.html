Robustness — Error Handling Tour

A tour of JavaScript's error-handling toolkit, applied to a UI: explicit idle → loading → success | error states, retries, real timeouts via AbortController, a custom TimeoutError class for typed branching in catch, and the two "last-resort" global handlers — error for uncaught synchronous throws and unhandledrejection for promise rejections that nothing handled. Open the console and try the two red buttons to see those last-resort handlers fire.

The error and timeout scenarios make real network requests to httpbin.org: error hits a 500 endpoint (fetch resolves but res.ok is false — a common gotcha), and timeout hits a 10-second slow endpoint with a 3-second client-side timeout that aborts the request when it loses the race.

About the two red buttons

The user-card scenarios above each handle their own errors locally with try/catch — that's the first line of defense. The two red buttons in the test bench are deliberately broken: one throws synchronously with no surrounding try, the other rejects a promise with no .catch(). They skip the first line on purpose.

The global listeners in scripts/main.js are the last line of defense: window.error catches the sync throw, window.unhandledrejection catches the promise rejection — two separate events because async failures settle after the call stack is gone, so the regular error event can't see them. Click each button and watch the page paint a coloured banner: that banner is your listener doing what would be a Sentry upload in production.

What's next — the testability problem

Notice that <user-card> calls fetch() directly inside fetchWithTimeout. That's fine for the demo, but it raises a testing problem: how do you write a test for the 3-second timeout path without actually waiting 3 seconds — or for the 500 path without hitting httpbin.org?

Lesson 08 (Dependency Injection) already showed the answer. The same pattern that let us swap memory and localStorage backends lets us swap a real fetch for a fake one in tests. Inject fetch (or a userService) rather than calling it directly — that's the testability win.
